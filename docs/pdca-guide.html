LOOP-SEED · PDCA WORKFLOW GUIDE

/pdca 使用教程与工作流程

从零讲起：这套机制解决什么问题、四个角色怎么配合、一条命令怎么用、以及怎么装到你自己的项目里。没接触过 loop 机制也能看懂。

§01

先搞懂这是什么（零基础）

如果你从没用过"loop 机制"，先读这一节。看懂了再往下，后面全是细节。

它想解决的问题

让 AI 改代码，最大的坑是它可能"自说自话"：改完直接说"好了，测试通过"——但它到底跑没跑测试？会不会为了让测试变绿，偷偷把测试删了、把断言注释了？你没法确认。

它的办法：把"改"和"验"拆开，互不信任

这套机制把一次代码修改拆成四个独立角色，每个角色权限被死死限制住，谁也没法既当运动员又当裁判。用工地打个比方就很清楚：

相当于 · 设计师
planner

先画图纸（方案），定清楚要改哪、改成什么样、怎么算完工。不亲自施工。

相当于 · 施工队
builder

只照图纸施工（写代码）。图纸没说的不擅自改，图纸和现场矛盾就停工上报。

相当于 · 监理
checker

只验收、只报问题。手里根本没有动工地的工具（没分配改文件权限），想造假也做不到。

相当于 · 项目经理
analyst

验收不过时拍板：是施工没照图纸（返工），还是图纸本身错了（改设计），还是这活超纲了（叫停）。

关键在于 checker 物理上无法改代码——它的工具清单里就没有"写文件"这一项。所以"测试通过"这句话不可能是它编出来的，必须是真跑出来的。这就是整套机制的信任基础。

"有界(bounded)"是什么意思

就是最多来回几轮就必须停（默认 5 轮）。防止 AI 卡在一个问题上无限次瞎试、把时间和额度烧光。到点了、或者发现"越改越糟""连续两轮同一个错"这类信号，机制会强制刹车，把情况原样交还给你决定，而不是硬撑。

一句话总结这是一个"防 AI 自说自话"的施工—监理机制：改代码的不负责验收，验收的改不了代码，出了问题有人专门判断该返工还是改方案，而且最多来回 5 轮就必须给你交代。
§02

快速开始（在本仓库里用）

本仓库（loop-seed）已经配好这套机制。在 Claude Code 会话里输入一条命令即可，剩下的流转全自动：

> /pdca 修复登录接口在密码含中文时返回 500 的问题

执行过程中你会看到类似这样的进度输出：

# 任务简报：目标 / 涉及区域 / 完成标准 / 要跑的检查
# planner 方案 v1：改法、涉及文件、完成标准、风险

Cycle 1/5
├─ builder  改了什么 / 修改文件 / 与方案的偏差
└─ checker  FAILED（2 个失败项 + 原始报错）
   analyst  FIX → 实现偏差，给出修复指引

Cycle 2/5
├─ builder  按指引修复
└─ checker  ALL GREEN（逐项通过证明）

✔ 完成：展示 diff 摘要 + 检查结果
注意命令是 /pdca 不是 /loop/loop 是 Claude Code 的内置定时循环命令（按固定间隔重复跑任务），和本框架无关。本框架的命令特意叫 /pdca 就是为了不撞名。
§03

四个角色的职责与权限

每个角色是一个独立子代理，在自己的上下文里运行。"不能改文件"由工具白名单强制——没分配 Write/Edit 就物理写不了，不靠自觉。

planner
Plan · 方案

基于实际读过的代码出方案：改法、涉及文件、可机械验证的完成标准。REPLAN 时修订方案并列出差异。

Read Glob Grep Bash（只读）
builder
Do · 实现

严格按方案实现，不扩大范围；发现方案与代码冲突时停下汇报，不自作主张。汇报必含"与方案的偏差"。

Read Write Edit MultiEdit Glob Grep Bash
checker
Check · 验收

按 CHECKS.md 选最小检查集运行，只输出三种状态之一：ALL GREEN / FAILED / BLOCKED，报错原文照抄不转述。

Read Glob Grep Bash（禁副作用命令）
analyst
Act · 反馈

判断失败出在哪一层：实现偏差→FIX，方案假设错→REPLAN，超能力边界→STOP。分析只能附加在原始报告旁。

Read Glob Grep Bash（只读）

§05

停止规则速查

由调度器机械执行，analyst 只能建议提前停，不能让循环多跑一轮。完整定义见 .claude/LOOP_RULES.md。

#条件说明
1ALL GREEN唯一的成功出口，须附每项检查的通过证明
2轮次用尽默认 5 轮上限，项目 CLAUDE.md 可覆盖
3同一失败 ×2按「检查命令 + file + 错误类型」归一化比较，不比报错原文；连续两轮同 key 即停
4出现回归之前通过的检查在后续轮次失败
5无实质进展连续 2 轮失败 key 数量没有减少；新失败不能抵消旧失败
6BLOCKED缺依赖/服务/凭证/环境，出现一次即停，不计入失败计数
7范围失控修复需要架构、schema、认证授权或生产部署级变更
8REPLAN 超限全任务最多重新规划 1 次，第二次 REPLAN 视为方案反复推翻，停止升级
§06

单独调用某个角色

不想跑完整循环时，可以直接点名子代理，适合"只看方案"或"只查状态"的场景：

> 用 planner 代理给"发票列表加导出 Excel"出个方案，先不要实现
> 用 checker 代理跑一遍后端相关检查，告诉我现在什么状态
> 用 analyst 代理分析一下这份测试报告，判断是实现问题还是方案问题
小任务别走 /pdca改 typo、调一行配置这种，四角色每轮多两次子代理调用，token 和时间成本都更高。直接让主会话改完、再单独派 checker 验一次，往往更划算。
§07

装到你自己的项目

这是最容易踩坑的一节，务必读完。把机制搬到别的项目，只需要复制几个文件——但有一个文件必须为新项目改写，否则命令能出现却验收不了任何东西。

第一步：复制哪些文件

把本仓库的以下内容整体复制到你项目的根目录：

文件 / 目录层要改吗作用
.claude/agents/（4 个角色）框架否四个子代理，命令派活的对象
.claude/commands/pdca.md框架否定义 /pdca 命令本身
.claude/LOOP_RULES.md框架否角色分工、停止规则、红线（与技术栈无关）
.claude/CHECKS.md项目必改本项目的技术栈、目录、检查命令
CLAUDE.md项目必改项目背景（Project Context 部分）
⚠ 为什么 CHECKS.md 一定要重写现在的 .claude/CHECKS.md 写的是本项目的检查命令（pytest tests/unit、npm run test:unit 等）。如果你把它原样搬到一个纯前端 / Go / Rust 项目，checker 去执行 pytest 会找不到，直接报 BLOCKED——循环跑起来了，但验收环节是空的，等于白跑。checker 和 builder 都是从这个文件里"发现"该跑什么检查的，所以它必须换成目标项目自己的命令。

第二步：为新项目改写 CHECKS.md

把 CHECKS.md 里三块内容换成你项目的实际情况：

  • 技术栈与目录：你项目用什么语言/框架，代码和测试放在哪
  • 检查命令发现顺序：checker 该读哪些配置文件找命令（如 package.json、Makefile、go.mod）
  • 默认检查集：不同改动范围分别跑什么命令（单测、构建、lint、类型检查等），且必须是只读无副作用的

第三步：改 CLAUDE.md 的项目背景

把 Project Context 换成新项目的一句话介绍，保留指向 .claude/LOOP_RULES.md 和 .claude/CHECKS.md 的说明即可。轮次上限想改（默认 5）也在这里覆盖。

第四步：确保是 git 仓库

如果项目还不是 git 仓库，先 git init 并提交一次。循环的可回滚性依赖每轮改动都有提交点——出问题能退回来。

第五步：验证

  1. 重开一个 Claude Code 会话，让它重新扫描 .claude/ 目录（新命令/新代理有时需要新会话才出现）。
  2. 输入 /pdca，确认命令出现在补全列表里。
  3. 先拿一个小而明确、预期会失败一两次的任务试跑一轮，观察 planner→builder→checker→analyst 的流转和停止规则是否正常，再用于正式任务。
设计初衷框架层（agents + pdca.md + LOOP_RULES.md）与项目层（CHECKS.md + CLAUDE.md）当初就是特意拆开的，就是为了让搬家时只动两个项目层文件，框架层一个字都不用改。
§08

常见问题

只放 .claude 文件夹和 CLAUDE.md 就能用 /pdca 吗？

命令会出现——但要真正跑通，必须先把 .claude/CHECKS.md 换成目标项目自己的检查命令，否则 checker 一步就 BLOCKED。详见 §07。

循环用哪个模型？

所有角色都 model: inherit，跟随主会话——你 /model 切到什么，整条循环就用什么。想让某个角色固定用更强/更便宜的模型，改对应 agent 文件的 model 字段即可。

循环中途我能插手吗？

能。循环在主会话里调度，随时可以打断补充信息或叫停。停止升级后修完阻塞点，直接再发一次 /pdca 重新开始（轮次从 1 重计，planner 会基于当前代码状态出新方案）。

为什么 checker 报 BLOCKED 而不是想办法装依赖？

设计如此：checker 无权改变环境（装依赖、启服务都算副作用），环境问题该由你决策。这防止循环在"检查环境"这件事上自作主张越走越远。

本仓库现在能直接跑通吗？

框架已就绪，但本仓库还没有业务代码——现在跑 /pdca 会在 checker 一步 BLOCKED（找不到可运行的检查）。等 backend/frontend/tests 就位后，先拿小任务试跑一轮。
